# /overall-bug-dashboard Workflow (v2.0)

> Single-view dashboard across ALL Winbull modules — bug status, sprint progress, developer attribution, fix velocity.
> Purpose: Management reporting, sprint reviews, weekly standups, or "where does the entire project stand?"
> Differs from: `/sprint-status` (sprint-level only), `/system-health` (risk heatmap, no developer attribution)

## Usage
Say: `/overall-bug-dashboard`

---

## Prerequisites
- `brain/winbull/7_bugs_and_issues.md` is up to date
- Optionally: `brain/winbull/_SYSTEM/FIX_VELOCITY.md` (for velocity trends)
- Optionally: GitHub tracking for live reconciliation

---

## Step 1: Load All Bug Data

Read `brain/winbull/7_bugs_and_issues.md`.

Extract per bug:
- Bug ID
- Module
- Severity (P0/P1/P2/P3)
- Category
- Sprint assignment
- Status: ✅ Fixed / ⚠️ Applied (not verified) / 🔧 In Progress / ⏳ Queue / 🔴 Open
- GitHub Issue number (if recorded: `GitHub: #{N}`)

Also read `brain/winbull/_SYSTEM/ACTIVE_BUGS.md` if it exists — in-progress/blocked bugs.
Also read `brain/winbull/_SYSTEM/FIX_VELOCITY.md` → get timing data (if exists).

---

## Step 2: Pull Live GitHub Status (Optional)

> If GitHub MCP is available, pull live data for reconciliation.
> If unreachable, fall back to local status from Step 1.

### 2a. Get All Open Issues
```
mcp_github-mcp-server_list_issues(
  owner: "{REPO_OWNER}",
  repo: "{REPO_NAME}",
  state: "open",
  per_page: 100
)
```

### 2b. Get All Closed Issues
```
mcp_github-mcp-server_list_issues(
  owner: "{REPO_OWNER}",
  repo: "{REPO_NAME}",
  state: "closed",
  per_page: 100
)
```

### 2c. Reconcile GitHub vs Local
For each bug with a GitHub Issue number:
- **GitHub closed + local fixed** → ✅ Consistent
- **GitHub closed + local in-progress** → ⚠️ Local stale — update local to fixed
- **GitHub open + local fixed** → ⚠️ GitHub stale — flag for GitHub update
- **No GitHub issue** → 📝 Local-only bug (not yet pushed to GitHub)

---

## Step 3: Calculate Metrics

For each module, compute:
- Total bugs
- Count by severity (P0/P1/P2/P3)
- Count by status (Fixed/Applied/In Progress/Blocked/Open)
- % Done = (Fixed + Applied) / Total × 100

System-wide:
- Total bugs across all modules
- Overall fix rate %
- Average fix time (if FIX_VELOCITY.md has 5+ entries)

---

## Step 4: Build Developer Attribution Table

If GitHub data available from Step 2:
1. Group closed issues by who closed/fixed them
2. For each developer:
   - Count total bugs fixed
   - Count by severity (P0/P1/P2/P3)
   - Calculate average fix time (if FIX_VELOCITY.md has data)
3. For open/in-progress issues, show who is currently assigned

If no GitHub data: skip this section.

## Step 7: Present Highlights

Point out to user:
1. Any P0 bugs still open → **Action required NOW**
2. Any blocked bugs → need human decision
3. Applied-but-unverified bugs → need client confirmation
4. Any data inconsistencies between local and GitHub
5. Velocity trend — speeding up or slowing down?
6. Developer attribution — who's contributing most
